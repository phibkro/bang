K1the oracle speaks
A reference you can interrogate

The Lean reference is an executable oracle; the differential harness drives any candidate against it. The effect-row unifier is verified — rows as a Mathlib Finset semilattice, unify_sound standing — and oracle-checked across thousands of cases.

K2the machine falls out
The VM is calculated, not designed

Bahr–Hutton calculation derives (compile, Code, exec) from eval for the thunk + force + first-order-effect core. The instruction set is an output of the derivation, correct by construction. Lean compiles exec to C — the first runtime you own outright.

proves exec ∘ compile ≡ eval runs core BANG on an owned machine

change the underlying monad; carry the handler stack into the configuration

K3effects on the machine
The real language runs on it

The monadic calculation lands: the effect-row monad for effects, the partiality monad for divergence. The handler stack becomes machine configuration; effect operations become instructions; STM is axiomatized as machine primitives. Multi-shot handlers are deferred frontier, named not faked.

proves correctness over the effect monad runs effectful BANG, end to end

give the source a way in: parse, type, infer effects

K4the front end is real
You write source, it runs

Parser → typed AST → effect-row inference (on the verified unifier) → core IR feeding the calculated compiler. A BANG file type-checks with full effect inference and executes on the owned runtime. The thing stops being a kernel and starts being a language.

K5the fast path · optional
Speed, without losing the truth

An optimized bytecode interpreter (C/Rust/Zig) and/or a calculated lowering of Code to C, Wasm, or WasmFX stack switching. Effect TS, once the planned backend, becomes just one optional lowering here — no longer the foundation. All differential-tested against the verified exec, which stays the oracle. The slow correct machine never retires; it adjudicates.

proves lowering refines exec (or diff-tested) runs at UX-acceptable speed · portable

The canon — calculating correct compilers

read top to bottom · PDFs on pszgmh (cs.nott.ac.uk/~pszgmh) & bahr.io

1967McCarthy & Painter — Correctness of a Compiler for Arithmetic Expressionsthe origin. the first proof that a compiler is correct. everything here is a 50-year reply to it.
2004Hutton & Wright — Compiling Exceptions Correctly · Calculating an Exceptional Machinewhere calculate, don't verify begins — derive the machine by defunctionalization.
2015Bahr & Hutton — Calculating Correct Compilers · JFP 25the foundational method. compiler + machine fall out of one correctness equation. Coq-formalized. start here for the technique.
2016Hutton & Bahr — Cutting Out Continuationsthe continuation-passing intuition behind the calculations, in miniature.
2017Hutton & Bahr — Compiling a 50-Year Journey · JFP 27the retrospective tying McCarthy–Painter to the modern method. read it as the map.
2020Bahr & Hutton — Calculating Correct Compilers II: Return of the Register Machines · JFP 30same method, register targets instead of stack. proof that the technique generalizes.
2021Pickard & Hutton — Calculating Dependently-Typed Compilers · ICFPyour Lean blueprint. the calculation in a dependently-typed setting, intrinsically typed, easier to mechanize.
2022Bahr & Hutton — Monadic Compiler Calculation · ICFPthe hook for BANG's effects. divergence via the partiality monad; generalizes to any effect by swapping the underlying monad.
2023Bahr & Hutton — Calculating Compilers for Concurrency · ICFPthe concurrency extension — nearest the surface for BANG's fibers and STM.
2024Garby, Hutton & Bahr — Calculating Compilers Effectively · Haskell Symp.deriving compilers that emit efficient code, not just tree-shaped code.
2024Bahr & Hutton — Beyond Trees: Calculating Graph-Based Compilers · ICFPjumps and graph-structured code via HOAS — the shape real backends need.
2024Tsuyama, Cong, Masuhara, Keller & Wang — An Intrinsically Typed Compiler for Algebraic Effect Handlers · PEPMadjacent & vital — handlers compiled to a typed stack machine, in Agda. the closest existing artifact to K3.
—Geeson (Oxford MSc) — Calculating Compilers and Algebraic Effectsthe method × algebraic effects, including CBPV with exceptions. a worked precedent for the frontier frames.
